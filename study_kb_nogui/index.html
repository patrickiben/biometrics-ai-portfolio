Study Knowledge Base — no GUI REFERENCE SKELETON

Files your whole study's email into a searchable knowledge base automatically: a scheduled script reads your Outlook study folder and turns every new message into a note + rebuilds a searchable index. You set it up once; after that it runs hands-off. No Power Automate flow, no SharePoint clicking, no Copilot.

Status — not validated software. This is a paste-able script + operating skeleton, structure-verified (not executed) in authoring; run/parse it on your own Windows machine first. Reported or regulated numbers always come from validated tools — never from a note or an LLM summary of one.

This is the no-GUI equivalent of the old Copilot/Outlook wiki's "when a new email arrives in the study folder → file it" Power Automate flow. The difference: the trigger is a scheduled local script, not a cloud flow. Point it at your study mail once (an Outlook rule into a folder, or a subject/sender filter) and it files the entire study automatically — there is no flow to build, no SharePoint pages to click, and no AI Builder/Copilot credits. Dragging individual emails is only an ad-hoc fallback for the odd message that lives outside the study folder.

At a glance

The trigger

Automatic, whole-study

A scheduled run reads your Outlook study folder and files every new message itself — no per-email clicking. It remembers what it's filed, so it never double-files and only ever reads your mail. Runs while your Outlook is open and you're signed in; if it's closed it just catches up next run — nothing's lost.

The engine

One scheduled script

Update-StudyKB.ps1 reads Outlook directly, writes one clean HTML note per message, and rebuilds the index — no flow builder, no SharePoint clicks, no Copilot.

The result

A searchable wiki

One index.html you search instantly; an on-device assistant (no cloud) can answer questions over it. Operations-only — never protected health information (PHI).

How it works

study mail lands in an
Outlook folder (a one-time rule — or filter the Inbox) → Update-StudyKB.ps1
every ~15 min — reads the folder, files all new mail → Notes\*.html + index.html
rebuilt each run → OneDrive syncs to SharePoint (no API)

On each run the script connects to your already-open classic Outlook, reads the study folder(s) you named (or sweeps the Inbox by subject/sender), and files every message it hasn't filed before. It remembers what it's done in .state\seen.json (Outlook EntryIDs), so re-runs are idempotent and your mailbox is only ever read — nothing is moved, deleted, or marked. (EntryIDs are per-store, so mail you later move to an archive/PST can re-file as a harmless duplicate note — never data loss; the seen.json set is the idempotence key, so keep it.) OneDrive does the SharePoint upload, so there's no Graph/connector/API.

It needs classic Outlook for Windows, open and signed in, in your own session (that's how it reads mail with no API and no admin rights). It attaches to your running Outlook — it won't launch one; if Outlook is closed it simply skips auto-ingest and picks up on the next run. The "new" Outlook and Outlook on Mac don't expose this automation — for those, use the drop-folder path in the README. On the first run you may see Outlook's "a program is trying to access Outlook" prompt — it only reads; allow it (or, the clean enterprise path, have IT code-sign the script and trust that publisher in Trust Center ▸ Programmatic Access, so the guard stays on for everything else).
Where the data lives. The notes and index are plain files in your OneDrive/SharePoint-synced library — so the knowledge base inherits that library's permissions, DLP, and retention, and anyone who can read the library can read the whole study's filed mail. Point it at a restricted, ops-team-scoped SharePoint site (not a personal OneDrive, not an open Team site). Content stays inside your governed M365 tenant — no third-party API or connector — but it does sync to the cloud tenant by design.

Set it up once

About ten minutes, once — then it runs itself. The screens below match classic Outlook for Windows; your version may look slightly different, but the option names are the same. You'll do three things:

① put the script in the synced folder  ·  ② make one Outlook rule so study mail lands in a folder  ·  ③ schedule the script. Step ② is the only fiddly part, so it's drawn out screen by screen.

Before you start, you'll need: classic Outlook for Windows (desktop), kept open during the workday  ·  a synced SharePoint/OneDrive library folder on your PC (where the wiki will live)  ·  Windows PowerShell (built into Windows — nothing to install)  ·  about 10 minutes. No admin rights, no new software. (On new Outlook or a Mac? The rule path needs classic Outlook — see the note at the end of this section.)

Exceptions (the red ticks): sensitive channels and sensitive subject words are excluded — they stay in your Inbox and never reach the folder, so they're never filed.
8

Name it and turn it on. Type a name (e.g. CP-101 → KB), make sure Turn on this rule is ticked → Finish.

From now on, operational email lands in the CP-101 folder; sensitive channels stay in your Inbox. Shared mailbox or a deeper folder? When you schedule the script (next), pass the path, e.g. -StudyFolders "Project Mailbox\Inbox\CP-101"; separate several with ;.

What the rule can — and can't — do. An Outlook rule can't read an email and decide whether it's "participant-level" — it only matches who sent it, where it came from, and keywords. So you keep participant data out by routing on source (your bounded set of ops senders/distros) and excepting the sensitive channels — not with a "subject contains CP-101" rule, which would route a participant query straight in. The rule is a coarse pre-filter; the real assurance is layered — you review what lands, the KB is ops-only by policy, and the number-validator guards anything drafted from it. Scope: this is built for a blinded operational mailbox. Sensitive channels (lab portals, safety/PV, IVRS) can still reach a blinded inbox — which is exactly why routing is by source + an explicit except-list — and if an operator receives unblinded data by role, source-routing alone is insufficient and the KB shouldn't auto-ingest their Inbox.

No folder rule possible? You can instead point the script straight at your Inbox by sender: add -SenderContains "cro-bioanalytics" (a known ops distro/domain) when you schedule it. Prefer that over -SubjectContains "CP-101" — a subject tag also matches a participant email about CP-101. This path has no exceptions, so use it only when study mail comes from a clean, known sender.

③ Make the script run automatically

9

Schedule it to run every ~15 min while you're logged on. Open PowerShell (Start menu → type "PowerShell" → Enter) and paste this — it's one command (the ^ just continues the line):

schtasks /Create /TN "StudyKB Update" /SC MINUTE /MO 15 /IT /F ^
  /TR "powershell.exe -NoProfile -ExecutionPolicy Bypass -File \"C:\Users\you\Study Knowledge Base\Update-StudyKB.ps1\" -StudyFolders CP-101"

Change C:\Users\you\Study Knowledge Base to your real folder from step 1. /IT = "interactive" — it runs on your desktop next to your open Outlook. Don't run it as SYSTEM or a stored-password task (there's no Outlook in that session, so it can't attach).

For IT (preferred posture): code-sign the script or allowlist its path; -ExecutionPolicy Bypass above is the unsigned quick-start — ExecutionPolicy is an execution-convenience setting, not a security boundary (per Microsoft). The auditable controls are the signed/allowlisted path and the read-only Outlook access.

10

Test it once now. In the same PowerShell window, paste this and watch it back-fill the last 30 days:

powershell -NoProfile -ExecutionPolicy Bypass -File "C:\Users\you\Study Knowledge Base\Update-StudyKB.ps1" -StudyFolders CP-101

On the very first run you may get Outlook's "a program is trying to access Outlook" prompt — it only reads; click Allow. After this you don't touch it — while Outlook is open the scheduled job files new study mail on its own.

Ad-hoc capture (the exception, not the routine)

Almost everything files itself through the rule above. This is only for a one-off that never reaches the study folder — e.g. a thread someone forwards you from outside the study. Operations content only — never drop a participant-level, unblinded, or sponsor-restricted email in here. You put the email in the _Inbox folder and the next scheduled run files it. Ask the knowledge base with a local model OPTIONAL · LATER PHASE

Not part of the core skeleton — the deliverable above is a deterministic file-mover with no AI. This is a later add-on; skip it in a first demo. That said: Notes\ is a small, clean corpus — good for an on-device assistant so you can ask questions over the whole study without any cloud. The flow: strip the HTML tags → embed → ask, all locally. A minimal recipe:

  1. Plain-text the notes (one .txt per note). PowerShell, no extra tools: 
    New-Item -Type Directory -Force "$Root\.rag" | Out-Null
Get-ChildItem "$Root\Notes\*.html" | ForEach-Object {
  $t = (Get-Content -Raw $_) -replace '<[^>]+>',' ' -replace '&nbsp;',' ' -replace '\s+',' '
  Set-Content "$Root\.rag\$($_.BaseName).txt" $t.Trim() -Encoding UTF8
}
  2. Point your local model at .rag\ — the same on-prem GGUF used in tlf_interpret (llama.cpp / its server, with a small local embedding model). Retrieve the top notes for the question, pass them as context, and require the answer to quote and cite the note it used.
  3. Keep it grounded: the model may only restate what's in the notes; a reported or regulated number is never sourced from a note or its summary — it comes from the validated system. (The notes already de-duplicate: one note per message, by EntryID.)
The local model runs on-device — no content goes to any third-party AI or external API. (The KB files themselves live in your governed SharePoint tenant — see Where the data lives.) This is the "ask the study its own history" win, governed: ops content in, restated-with-citation out, numbers still owned by the validated tools.

If a script won't run (IT policy)

Preferred (production posture): have IT code-sign the script — or allowlist its path — which also satisfies change control. -ExecutionPolicy Bypass is the unsigned quick-start; note that ExecutionPolicy is an execution-convenience setting, not a security boundary (per Microsoft), so the signature/allowlist is the real control. If a run is blocked: Unblock-File .\Update-StudyKB.ps1 first; if still blocked, ask IT for the signature/allowlist. If your org blocks programmatic access to Outlook (the prompt never clears), the clean fix is to code-sign the script and trust that publisher in Trust Center ▸ Programmatic Access — keeping the object-model guard ON for everything unsigned — or fall back to the drop-folder path. No admin rights are otherwise needed.

Governance (unchanged)

Operations content only — no PHI, no participant-level data, no unblinded/sponsor-restricted material in the knowledge base. The auto-ingest files whatever reaches the study folder, and an Outlook rule can only filter by source and keywords — not by understanding content. So keep participant data out with a layered control, not one clever rule: (1) you review what's filed, and the KB is ops-only by policy; (2) route the folder from your operational senders/distros and except the sensitive channels (safety/PV, EDC/IVRS/randomization, unblinded-stats, lab portals); (3) it assumes a blinded operational mailbox — sensitive channels can reach a blinded inbox, so the except-list is a precondition, not a nicety; if the operator is unblinded by role, don't auto-ingest their Inbox. Reported or regulated numbers always come from validated tools, never from notes or an LLM summary of them.

What we're asking for

Not budget, not a new system, not production use — just approval to run a small, governed pilot:

  • Scope: this tool, on one operator's blinded ops mailbox, one study — starting against a test folder. Operations content only.
  • Footprint: nothing new to buy or stand up — it runs on the Microsoft 365 tenant and classic Outlook the team already has.
  • Governance: IT signs/allowlists the script; a restricted, ops-scoped SharePoint site; a named owner reviews what's filed; validated tools still own every reported number.
  • Decision: a go/no-go gate at the end of the window, judged on three things — the team can find the study's operational history, nothing participant-level ever lands in it, and no data-residency surprises.

It's the lowest-risk piece of the program — governed, reversible, and self-contained — which makes it the right place to prove the operating pattern (pilot → gate → decide) that the rest of the program follows. The wins it's after are shared team memory (the study's history survives someone being on leave), consistency (one searchable record instead of scattered inboxes), and speed of recall — with sensitive data kept in-tenant.
